# Environment variables (full operator reference)

Every tuning flag, kill switch, and opt-in the plugin honors. Each entry states
the default, what the flag changes, and the design contract behind it (default-ON
with kill switch for offline features; opt-in only for repo-code execution or
network). Keep this file in sync when adding or changing a flag.

- `CHAMELEON_DISABLE=1` ŌĆö disable plugin globally for this session
- `CHAMELEON_VERIFY=0` ŌĆö disable PostToolUse archetype verification (default ON)
- `CHAMELEON_ENFORCE=0` ŌĆö kill switch for all blocking enforcement (PreToolUse deny, PostToolUse block, Stop backstop). Disables blocking regardless of `enforcement.mode`. NOT the same as full silence at every hook: PreToolUse still injects its per-edit pattern advisory (it just never denies), but the Stop backstop returns nothing ŌĆö its turn-end advisories (deterministic checks like cross-file existence, and the review job's correctness/duplication/idiom lens findings) are suppressed along with its block, so ENFORCE=0 is "advisory-only" at the per-edit hooks and fully silent at Stop. For advisory-only turn-end review WITHOUT the block, set `enforcement.mode=shadow` (or `off`) instead, which keeps the Stop advisory pipeline running. Blocking otherwise follows `.chameleon/config.json` `enforcement.mode`: `off` = advisory only, `shadow` = log would-have-blocked but never block, `enforce` = real deny/block on calibrated rules (default). A blocked edit is overridable inline with `// chameleon-ignore <rule>` (`# chameleon-ignore <rule>` in Ruby/Python).
- `CHAMELEON_ALLOW_ESLINT_EVAL=1` ŌĆö opt into loading JS ESLint configs via Node `require()`/`import()` during bootstrap (default OFF; off uses a static parser that never executes repo code). Enable only for repos you trust.
- `CHAMELEON_ALLOW_DEP_AUDIT=1` - opt into the `dep_audit` MCP tool, which shells the repo's own `npm audit` / `bundler-audit` (default OFF). Off refuses rather than spawning a network process behind your back. This is the only supply-chain check that touches the network and runs tool-time only, never on a hook hot path; it fails open to an "unavailable" result when the binary, manifest, or network is absent. The no-network manifest/lockfile diff checks in the pr-review skill run regardless of this flag.
- `CHAMELEON_ALLOW_TSC=1` ŌĆö opt into the auto-pass router's `tsc --noEmit` grounding run (default OFF). Executes the repo's own tsc binary, resolved exclusively from `<repo>/node_modules/.bin` (never PATH, never a download), with a hard timeout. Tool-time only, never on a hook hot path. Off (or no root tsconfig.json, or no installed tsc) reads as a recorded "typecheck unavailable" fact ŌĆö it never blocks auto-pass eligibility on its own.
- `CHAMELEON_ALLOW_TESTS=1` - opt into the auto-pass router's repo-local test run (default OFF). The sibling of `CHAMELEON_ALLOW_TSC`: resolves vitest/jest exclusively from `<repo>/node_modules/.bin` (never PATH, never npx, never a download), with a hard timeout (default 300s). Tool-time only, never on a hook hot path. Off reads as a recorded "tests unavailable" fact; only actual test failures route a change to a human, never the unavailable state.
- `CHAMELEON_JUDGE_ASYNC=1` ŌĆö DEPRECATED, now a no-op. Pre-3.4.0 this opted into a detached post-Stop correctness-judge spawn; since the async-first cutover EVERY review job runs detached by construction (there is no synchronous judge path left to opt out of), and nothing in `stop/scheduler.py` or `stop/job.py` reads this variable ŌĆö setting it changes nothing. Kept documented only so an operator who set it under the old pipeline understands it is now inert. Use `CHAMELEON_JUDGE_WAIT=1` if the goal is in-turn (rather than next-turn) findings delivery.
- `CHAMELEON_JUDGE_WAIT=1` ŌĆö the harness/eval synchronous-wait path over the async-first detached review job (default OFF; spec "Stop hook + idiom system overhaul" section 3.1). A one-shot `claude -p` session (the effectiveness eval, the journey harness, CI) has no next UserPromptSubmit to deliver a detached job's findings into, so under async-first review a model-reviewed finding would never reach that session's own transcript. With the flag set, after the scheduler would normally launch a review job and return, Stop instead polls for that SAME job's completion (bounded by the Stop hook's own remaining budget) and renders whatever is ready in-turn, trading immediacy for a one-shot session's only chance to observe review output. Polling ends on whichever comes first: the job's session-doc slot clears (a clean exit) or its heartbeat goes stale (a crash ŌĆö the identical staleness window `scheduler.try_acquire_job_slot` itself reclaims on), or the budget expires (nothing to show this turn; the finding stays `pending` for the next real delivery point, exactly like an ordinary async turn ŌĆö never a drop). Once done, the job's own pre-rendered delivery payload is preferred (freshest, cheapest); a live ledger render is the fallback. Wired into the live Stop pipeline (`stop/pipeline.py`'s `_run_review_job`, called from `stop_gates`): after a successful `scheduler.launch_job`, this flag decides whether the same Stop then polls; the poll's own `TurnBudget` (`JUDGE_WAIT_STOP_BUDGET_SECONDS`, default 35s, leaving headroom under the 55s wrapper cap) is separate from the job's own generous budget. See `wait_and_render` in `plugin/mcp/chameleon_mcp/stop/judge_wait.py`. The journey harness (`tests/journey/harness/context.py`) and the effectiveness eval's base run env set this so a one-shot `claude -p` session observes findings in-turn instead of never seeing them.
- `CHAMELEON_JOB_TOTAL_BUDGET_SECONDS` / `CHAMELEON_JOB_LENS_BUDGET_SECONDS` / `CHAMELEON_JOB_VERIFY_BUDGET_SECONDS` / `CHAMELEON_JOB_RENDER_BUDGET_SECONDS` ŌĆö the detached review job's total wall-clock budget (default 240s) and its per-stage advisory shares (lenses 150s, VERIFY 60s, render 10s; a later miner stage reserves the remainder). The job runs off the Stop hook's own clock ŌĆö it is spawned detached and outlives the hook process ŌĆö so this is generous relative to the old 45s synchronous judge budget. The shares are splits a stage MAY use to size its own sub-timeouts, not a hard per-stage kill; they sum to less than the total so the runner keeps slack for process startup and the final write. See `JOB_TOTAL_BUDGET_SECONDS` etc. in `plugin/mcp/chameleon_mcp/_thresholds.py` and `stop/job.py`.
- `CHAMELEON_JOB_HEARTBEAT_INTERVAL_SECONDS` / `CHAMELEON_JOB_HEARTBEAT_STALE_SECONDS` ŌĆö how often the detached job refreshes its heartbeat file's mtime while it runs (default 10s) and how old a heartbeat may get before the single-inflight slot is reclaimed as a dead job (default 30s). The stale threshold sits comfortably above the write interval so one missed tick under load never reads as dead; `scheduler.try_acquire_job_slot` is the reclaim path. See `stop/scheduler.py` and `stop/job.py`.
- `CHAMELEON_IDIOM_LENS_MAX_IDIOMS` / `CHAMELEON_IDIOM_LENS_MAX_PROMPT_BYTES` / `CHAMELEON_IDIOM_LENS_MAX_FINDINGS` ŌĆö the idiom lens's reviewer prompt economy (default 12 idioms / 60,000 prompt bytes / 8 findings per spawn), mirroring `DUPLICATION_REVIEW_MAX_FINDINGS`'s per-turn shape so one broad edit cannot flood a turn with idiom findings. See `stop/lenses/idiom.py`.
- `CHAMELEON_REVIEW_RENDER_TOKEN_CEILING` ŌĆö `stop/assemble.py`'s greedy-pack ceiling for one delivery emission (UserPromptSubmit or in-turn `CHAMELEON_JUDGE_WAIT`), across every discovered workspace root (spec section 6: "One emission" ŌĆö not per-root); default 800 tokens. An item that does not fit whole is omitted from the render and stays `pending` for the next delivery point ŌĆö never truncated mid-item. `CHAMELEON_SESSION_START_DELIVERY_TOKEN_CEILING` is the wider counterpart (default 2,500) for SessionStart's dead-session delivery, which has no 3s wrapper cap to protect. See `stop/assemble.py`.
- `CHAMELEON_STOP_VERIFY=0` ŌĆö kill switch for the turn-end VERIFY stage (default ON). Runs inside the detached review job, after every active lens (`stop/verify.py::verify_findings`, called from `stop/job.py`): the independent refuter (`refuter.py`, the same engine the pr-review round-3 refuter uses) checks each REFUTABLE finding (kind `correctness` or `idiom` ŌĆö a `duplication` finding's evidence spans two locations a one-file excerpt cannot show, so those pass through pre-confirmed instead) before it surfaces ŌĆö a refuted finding is DROPPED, a confirmed one is tagged `[confirmed]`, everything else surfaces `[unverified]`. Strictly raise-precision: a refuted finding is the ONLY thing it removes; on any failure (disabled, no budget, CLI absent, spawn error, no verifiable excerpt) it passes every finding through unverified; and a finding whose excerpt cannot be safely fetched (no file/line, unreadable, or a path escaping the repo ŌĆö containment-checked via `safe_read_text`, since the file field is model output inlined into a prompt) is NEVER spawned. Budget-honest: spawns are retry-free (`refuter.run_batch(..., retry=False)`) so a slot is bounded by exactly one timeout window, sized against the job's own remaining `JOB_VERIFY_BUDGET_SECONDS` share, high-severity findings first when the budget cannot cover every spawnable one. Off the per-edit hot path (job-only), no repo-code execution and no network (the refuter is a hardened no-tools `claude -p`), so it follows the default-ON-with-kill-switch principle. Refuter model/timeout/spawn-cap reuse `CHAMELEON_REFUTER_MODEL` / `REFUTER_TIMEOUT_SECONDS` / `REFUTER_MAX_SPAWNS_PER_INVOCATION`. See `verify_findings` in `plugin/mcp/chameleon_mcp/stop/verify.py`. (The pre-3.4.0 `stop_verify.py::verify_stop_findings` this entry used to describe has no live caller left ŌĆö `_contained_rel`/`_excerpt_window` are its only still-used pieces, imported directly by `hook_helper._pending_findings_block` and the co-change advisory.)
- `CHAMELEON_JUDGE_MODEL` ŌĆö model the turn-end correctness judge and duplication/idiom job lenses spawn (`claude -p --model <value>`); default `sonnet`. Lower to a cheaper model or raise for a stronger reviewer; affects only the advisory turn-end review spawns, never the hook hot path.
- `CHAMELEON_JUDGE_MODEL_HIGH` ŌĆö model the correctness judge escalates to on a HIGH-risk route (`risk_high` / `intent_forced`), per the reviewer model ladder (roadmap #6, `judge_model_for_route` in `plugin/mcp/chameleon_mcp/judge.py`); default `opus`. Every other route uses `CHAMELEON_JUDGE_MODEL` (default `sonnet`). Unconditional since the async-first cutover: every review this ladder routes for now runs inside the detached job (`stop/job.py`), which owns its own generous wall-clock budget, so there is no separate synchronous path left to protect from a slower model ŌĆö the pre-3.4.0 ladder's `_RUNNING_DETACHED` gate (which never actually fired post-cutover, since nothing called `judge.mark_detached_run()`) has been removed from `judge_model_for_route`. Raise-only and never garbage: an unrecognized HIGH model falls back to the (valid) base rather than spawning a model id that would fail-open the judge to zero findings. `CHAMELEON_JUDGE_TIERING=0` flattens every route to the base model. `judge._RUNNING_DETACHED` / `judge.mark_detached_run()` still exist for one unrelated purpose: `_reviewer_timeout_seconds`'s fallback budget for the one reviewer spawn that does not thread its own timeout (`duplication_review.judge_body_matches`, called from `stop/lenses/duplication.py`). That fallback still never takes the generous branch today ŌĆö the same zero-caller gap ŌĆö a separate, low-impact follow-up (wiring `mark_detached_run()` into `stop/job.py`'s entry point) left open.
- `CHAMELEON_JUDGE_TIERING=0` ŌĆö kill switch for the reviewer model ladder (default ON). Flattens the correctness judge AND the refuter to their base models (`CHAMELEON_JUDGE_MODEL` / `CHAMELEON_REFUTER_MODEL`) on every route/severity, restoring the pre-#6 flat-model behavior.
- `CHAMELEON_REFUTER_MODEL_HIGH` ŌĆö model the round-3 refuter escalates to for a BLOCK / high / critical-severity finding; default `opus`. Nit/FIX-severity findings keep `CHAMELEON_REFUTER_MODEL`. Same raise-only / never-garbage guard and `CHAMELEON_JUDGE_TIERING=0` kill switch as the judge ladder. See `_refuter_model_for` in `plugin/mcp/chameleon_mcp/refuter.py`.
- `CHAMELEON_DUP_MODEL` ŌĆö model the turn-end duplication confirm spawn uses (`claude -p --model <value>`); default `sonnet` (unchanged from when it rode the judge's sonnet default), now independently tunable. An unrecognized value falls back to `sonnet` rather than fail-opening the spawn.
- `CHAMELEON_REVIEW_REFUTER=0` ŌĆö disable the pr-review / receiving round-3 refuter (the independent `refute_finding` spawn); the skills fall back to inline verification. Default ON.
- `CHAMELEON_REVIEW_FANOUT=0` ŌĆö disable pr-review large-diff fan-out; review runs single-pass inline. Default ON.
- `CHAMELEON_REFUTER_MODEL` ŌĆö model for the round-3 refuter spawn (default `sonnet`); same role as `CHAMELEON_JUDGE_MODEL` for the turn-end judge.
- `CHAMELEON_FETCH_PRODUCTION_REF=0` ŌĆö kill switch for the default-ON production-ref fetch. When a repo has a locked `production_ref`, refresh (manual `/chameleon-refresh` AND the auto-refresh) runs one bounded, non-interactive `git fetch origin <branch>` BEFORE resolving the tip, so derivation sees the genuinely-latest production instead of the user's last fetch. This is the one network path made default-ON (the per-repo flag is `auto_refresh.fetch_production_ref`, default true). It self-suppresses under `CI` (a fresh CI clone must not do an unasked network fetch), never runs on a hook hot path (PreToolUse/PostToolUse/SessionStart stay offline), and fails open to the last-fetched ref with a classified reason surfaced in the refresh envelope + `auto_refresh.log`. Tuning: `CHAMELEON_PRODUCTION_REF_FETCH_TIMEOUT_SECONDS` (default 10; SIGKILLs a stuck transfer) and `CHAMELEON_PRODUCTION_REF_FETCH_BACKOFF_HOURS` (default 6; after an auth/branch-gone failure, retry only this often).
- `CHAMELEON_INTENT_CAPTURE=0` ŌĆö kill switch for UserPromptSubmit intent capture (default ON). Capture persists only hard-secret-scanned extracted assertion tokens and content digests, never raw prompt prose; the captured tokens force the correctness-judge security/intent lens when they contain checkable constants.
- `CHAMELEON_ATTESTATION=0` ŌĆö kill switch for the turn-end session attestation record (default ON; local-only writes, no network, no repo-code execution). The attestation is self-signed and raise-only: nothing in it may ever lower scrutiny, it exists to raise gate depth and make post-incident replay honest.
- `CHAMELEON_FINDING_LEDGER=0` ŌĆö kill switch for the turn-end resurface re-check (default ON; roadmap #9). Every finding a review-job lens persists lives in the canonical finding-lifecycle ledger (`review_ledger.py`'s `findings_ledger.json`, one JSON row per repo keyed by match_key; `stop/job.py` writes to it via `record_findings`). At the START of the next Stop, BEFORE that turn's own gates persist anything, `stop/pipeline.py`'s `stop_gates` calls `review_ledger.recheck_and_resurface` ŌĆö skipped entirely when this flag is `0` (checked at the call site, since the function itself is unconditional). Every open (`pending`/`delivered`/`resurfaced`) row is re-checked against the cited file's CURRENT content: changed or gone since review => `addressed` (dropped from the open set); unchanged and high/blocker severity => resurfaced exactly ONCE via a `<chameleon-context>` block, then marked `resurfaced` ŌĆö a TERMINAL status for ordinary delivery (`undelivered_findings` returns only `pending` rows, and `mark_delivered` refuses a `resurfaced` row as a source state), so the one-shot resurface line is the finding's SOLE re-appearance; it can never loop back through UserPromptSubmit/SessionStart delivery and re-arm itself into a repeat nag. Off the per-edit hot path (Stop only), fail-open to `[]`, bounded (`_RESURFACE_MAX_LINES`), rel_path + lens sanitized on the way to the model surface. Superseded a pre-3.4.0 drift.db-based store (`judge_findings` table, `record_judge_finding`/`open_judge_findings`/`mark_judge_finding` in `drift/observations.py`, `_ledger_persist`/`_ledger_recheck_and_resurface` in `stop/gates.py`) that the async-first cutover left wired to nothing (findings landed in the new ledger, but `stop/pipeline.py` still called the old one, which had nothing to read) ŌĆö that store and its functions have been retired. Any HIGH finding it was still holding open from before the cutover is not migrated (a documented, low-impact gap ŌĆö unlike the `.judge_pending.<session>.json` queue, which IS migrated via `review_ledger.migrate_pending_queue`). See `recheck_and_resurface` / `undelivered_findings` / `mark_delivered` in `plugin/mcp/chameleon_mcp/review_ledger.py`.
- `CHAMELEON_AUTOPASS_ATTESTATION=0` ŌĆö kill switch for attestation-gated auto-pass (default ON; roadmap #7). `get_autopass_verdict` reads the recent session attestations, attributes them to the branch diff by FILE OVERLAP (the `governed_files`/`ungoverned_files` of an attestation sharing a path with the diff; repo_id is already the ledger scope), and folds three governance signals in RAISE-ONLY: verification suppressed while the diff was written (`env.verify_off` / a `verify_env_off` skip, never the routine cooldown re-verify), the correctness judge spawn degraded (`degraded_spawn`, never the deliberate low-risk skip), or a chameleon-ignore override on one of the diff's own files. Each adds a soft (ŌåÆ elevated) needs-human reason via `classify_change`, so an under-governed diff routes to a human on terms a fully-governed one does not. Strictly raise-only per the attestation contract: no match leaves the verdict identical to today (a forged clean record buys nothing), and the router never blocks. Tool-time only (never a hook hot path), fail-open (any read error leaves the coverage all-clear). Scan window: `CHAMELEON_ATTESTATION_MATCH_LIMIT` (default 25 recent attestations). See `session_coverage_from_attestations` in `plugin/mcp/chameleon_mcp/autopass.py`.
- `CHAMELEON_TRUST_REVALIDATE=1` ŌĆö opt back IN to trust re-validation (default OFF). By default trust is ONE-TIME: once a repo is trusted it stays trusted across every later profile change (refresh, re-bootstrap, teach) and never goes "stale", so the user never re-grants. `=1` restores the old behavior where any change to the trust-hashed profile surface flips the grant to `stale` and re-prompts for a fresh `/chameleon-trust`. The staleness predicate is funnelled through `profile_diverged_from_grant` in `profile/trust.py` (read at call time), so all gates (detect_repo, the PreToolUse/PostToolUse/Stop enforcement gates, the statusline) honor it uniformly. Because persistent trust means a post-grant profile edit is never re-reviewed, the prose artifacts (`idioms.md`, `principles.md`) are injection/secret/dangerous-pattern scanned at their READ path (`loader._prose_injection_unsafe`, mtime-cached; the principles.md SessionStart read) and dropped if poisoned ŌĆö render-site sanitization does NOT neutralize injection prose, so this read-path scan is what closes the staleness-decoupled injection gap. See docs/architecture.md "Trust model".
- `CHAMELEON_MAX_CONVENTION_ITEMS` ŌĆö cap on each repo-size-scaling section of the SessionStart convention block (preferred imports, DSL calls, key-export union); default 60, over-cap shows a "+N more" tail. Raise to surface more, lower to shrink the block. Read at import time.
- `CHAMELEON_MAX_KEY_EXPORTS` ŌĆö cap on stored key exports per archetype (default 400). Read at import time.
- `CHAMELEON_MAX_SIBLINGS` ŌĆö cap on the per-edit "Nearby files" sibling listing (default 60). Read at import time.
- `CHAMELEON_NEARBY_SIGNATURES=0` ŌĆö kill switch for the per-edit "Nearby collaborator signatures" section (default ON). The section adds the real callable signatures (`name(param: type): ret ŌĆö path:line`) of source files in the edited file's directory to the Tier-2 block, read from the precomputed `symbol_signatures.json` (no live parse; mtime-cached), so the model sees the cross-file CONTRACTS it must call, not just sibling filenames. Candidates are ranked by call proximity: a sibling the edited file is recorded calling (from the reverse `calls_index.json`) leads, with deterministic name order as the tiebreak and the full order when no call facts exist. Bounded (5 files / 8 signatures / 700 chars; scored set capped by `CHAMELEON_NEARBY_SIG_SCAN_CAP`, default 200), fails open. Pure advisory, no repo-code execution and no network (only cached artifact reads), so it follows the default-on-with-kill-switch principle. A/B it with `--toggle nearby_signatures` (the paired arm sets `=0` to turn it off).
- `CHAMELEON_INBOUND_CALLERS=0` ŌĆö kill switch for the per-edit "Inbound callers of this file's exports" section (default ON). The counterpart to nearby-signatures: that shows OUTBOUND sibling contracts you might call; this shows INBOUND dependents that break if you change THIS file's exported signatures. On a Tier-2 Edit/Write, it reads the edited file's own exports (`symbol_signatures.json`) and, for each, the recorded call sites from the reverse `calls_index.json` (both mtime-cached, no live parse), rendering e.g. `getUser() <- src/order.ts:3, src/checkout/summary.ts:44 (+3 more)` with a "change a signature -> update these call sites in the same turn" directive ŌĆö converting chameleon's most-detected defect class (cross-file staleness, caught at turn end by the correctness judge) into pre-edit prevention. Rendered OUTSIDE the imitate-spotlight (a chameleon directive over repo-derived facts; paths/names sanitized at the boundary), next to the counterexample. Bounded (`CHAMELEON_INBOUND_CALLERS_MAX_EXPORTS`=3 exports / `CHAMELEON_INBOUND_CALLERS_MAX_SITES`=5 sites / `CHAMELEON_INBOUND_CALLERS_MAX_CHARS`=600 chars), fires only when real caller edges exist, and carries an honesty note (barrels and dynamic dispatch are invisible to the snapshot, so an empty list is NOT proof it's safe to break). No repo-code execution and no network (cached artifact reads only), so it follows the default-on-with-kill-switch principle; fails open to "". A/B it with `--toggle inbound_callers` (the paired arm sets `=0`). See `_inbound_contracts_section` in `plugin/mcp/chameleon_mcp/hook_helper.py`.
- `CHAMELEON_CROSSWS_INDEX=0` ŌĆö kill switch for the monorepo cross-workspace existence index (WP-C5, default ON). Closes the per-workspace reverse-index blind spot: `build_reverse_index` runs per workspace, so a file in package B importing from package A (a `@scope/a` cross-package edge) is absent from A's own index, and a removed export A's sibling still imports goes unseen. With the flag on, at bootstrap/refresh each workspace captures the cross-package import specifiers its own reverse index DROPS (in-memory on the BootstrapReport, never persisted per-workspace), the coordinator JOIN (in `_amend`'s successor after the workspace loop) resolves each `@scope/pkg` to the sibling workspace's file via a package.json-`name` map + fail-closed name-in-exports confirmation, and the resolved edges are written to a single `cross_reverse_index.json` in the PLUGIN DATA DIR (`~/.local/share/chameleon/<coordinator repo_id>/`) ŌĆö deliberately OFF the trust-hashed profile surface (a pre-code security review found that materializing a coordinator profile to host it would create a new trust anchor and arm the security-deny floor on previously-ungoverned root files). The consumer is a turn-end Stop ADVISORY (never a deny): for each edited TS file it resolves the coordinator index via the workspace profile's `workspace.parent.repo_id`, and flags a removed export a sibling workspace still imports, confirmed by a live presence re-check on the importer. TypeScript/JS cross-package resolution only in v1 (Python is a documented gap); off the per-edit hot path (bootstrap-time build + Stop-time read of a cached artifact); no repo-code execution and no network, so it follows the default-ON-with-kill-switch principle; fails open at every seam. See `_crossworkspace_existence_advisory_lines` in `plugin/mcp/chameleon_mcp/hook_helper.py` and `build_cross_reverse_index` / `load_cross_reverse_index` in `plugin/mcp/chameleon_mcp/symbol_index.py`.
- `CHAMELEON_ARCHETYPE_FACTS=0` ŌĆö kill switch for the per-edit archetype-scoped facts directive (default ON). On a Tier-2 (first-in-archetype) Edit/Write/NotebookEdit, the block leads with two compact directives scoped to the EDITED archetype: the class contract its files implement (base class, required methods, DSL macros ŌĆö e.g. a Rails ActiveInteraction service: extends `ActiveInteraction::Base`, define `execute`) and "Already defined in this archetype ŌĆö reuse these before creating a new one: ŌĆ”" (the archetype's own `key_exports`). Read from `conventions.json` (the same artifact the Tier-1 echo reads), scoped to one archetype, so it is additive over the repo-wide convention union injected once at SessionStart. Rendered OUTSIDE the imitate-spotlight (it is a chameleon directive, not witness data); every value is sanitized at the boundary. Bounded (8 methods / 8 macros / 40 exports with a `+N more` tail), fires only when the archetype actually has the data, no repo-code execution and no network (one cached artifact read), so it follows the default-ON-with-kill-switch principle. See `_archetype_facts_section` in `plugin/mcp/chameleon_mcp/hook_helper.py`.
- `CHAMELEON_COUNTEREXAMPLE=0` ŌĆö kill switch for the per-edit off-pattern counterexample (default ON). When a team has taught a competing import (`/chameleon-teach-competing-import`, "prefer X over Y") and a real file still uses the discouraged form, the Tier-2 block pairs the canonical witness with that captured off-pattern line as a "do NOT write it this way" directive ŌĆö the positive/negative contrast the in-context-learning literature favors over a positive example alone. When a team has taught SEVERAL competing imports for one archetype (winstonŌåÆlogger AND momentŌåÆdate), every still-present off-pattern is shown, not just the last taught (`counterexamples.json` is a list per archetype, schema v2; a legacy v1 single-row artifact still loads). The artifact (`counterexamples.json`, trust-hashed, drop-stale) is built at teach time (a bounded repo scan) and at bootstrap/refresh, never on a hook hot path; the edit-time read is mtime-cached and fails open. It fires only when a genuine taught off-pattern is still present, so a clean archetype injects nothing. Rendered OUTSIDE the imitate-spotlight (a counterexample must not be copied) but still sanitized. No repo-code execution and no network, so it follows the default-ON-with-kill-switch principle; set to `0` to disable. A/B it with `--toggle counterexample` (the paired arm turns it off).
- `CHAMELEON_STOP_IDIOM_TERSE=0` / `CHAMELEON_STOP_IDIOM_GIST=0` ŌĆö DEPRECATED, both now no-ops. These governed the pre-3.4.0 once-per-session Stop-hook idiom-review INTERRUPT (`_idiom_review_gate`, deleted in the async-first cutover) and its two renderers, `tools._render_stop_idioms` (terse scoping/summarizing) and the memory-channel gist layer. - `CHAMELEON_MEMORY_CHANNEL_DEDUP=0` ŌĆö kill switch for the SessionStart memory-channel dedup (default ON). When the repo already imports the conventions.md mirror (same delivery-faithful detection as `CHAMELEON_STOP_IDIOM_GIST`), the identical conventions content arrives through the HIGHER-authority memory channel at session load, so SessionStart drops the PORTION of its `<chameleon-conventions>` hook injection the mirror already carries and keeps only what it doesn't ŌĆö PER ITEM, not all-or-nothing (Phase 4; the pre-4 gate either kept the whole block or replaced it whole with a pointer). Lossless at LINE granularity: a bullet is dropped from the fresh injection ONLY when that exact line already appears in the text the import delivers AND the mirror also carries that bullet's section header (flat line membership alone is not enough, so a still-current rule recited out of section in a hand-authored "deprecated/rejected" decoy mirror does not suppress it), and a header stays attached to any bullet under it that survives rather than being emitted as an orphan bullet. A fully-covered block collapses to a one-line pointer (the historical full-dedup outcome); a pre-3.1.0 mirror missing the principles sections still injects those sections in full (the rest collapses to the pointer); a content-stale mirror (headers match but a newer rule line is absent) injects exactly that missing line plus the pointer, not the whole block; a mirror sharing nothing with the block, or a worktree whose import target is not materialized, keeps the full injection untouched (dedup bought nothing, so there is nothing to point at); any doubt or read error keeps the full injection too. The untrusted path is unchanged (still injects nothing). No repo-code execution and no network, so it follows the default-ON-with-kill-switch principle. Discovery globs `_plugin_data_dir()/*/.enforcement.<session-marker>.json` and regroups the recorded files by `find_repo_root(file)`; each workspace is gated with per-workspace trust (`grants_root`, never unioned, so an ungranted sibling under a monorepo-shared repo_id stays untrusted), at most ONE reviewer spawn is paid across the whole Stop (the ranked-first armed root owns the budget; the rest run deterministic-only), and the Stop short-circuits on the FIRST blocking root so the anti-loop `stop_block_cap` is charged to one root per Stop. Advisories from every non-blocking workspace merge into one Stop context; one signed attestation is written per distinct run-root. A degenerate empty/None session_id (marker `unknown`) skips the glob (cwd root only) so a shared bucket cannot pull unrelated repos in; the fan-out is bounded by `CHAMELEON_STOP_MAX_ROOTS` (default 16, armed roots ranked first so the cap only ever drops advisory-only roots). - `CHAMELEON_COCHANGE_HISTORY=0` ŌĆö kill switch for the framework-agnostic historical co-change signal (default ON). At bootstrap/refresh a single bounded `git log --name-only` walk mines files that change together across the repo's own history; a strong partner B of A (co-changed in >= `CHAMELEON_COCHANGE_HISTORY_MIN_RATIO` of A's commits, >= `CHAMELEON_COCHANGE_HISTORY_MIN_SUPPORT` commits) that a turn leaves untouched while editing A becomes a deterministic, zero-LLM turn-end omission advisory ("A usually changes with B; B is untouched"). Keys are work-tree-top-relative so a monorepo's workspaces share one global index; persisted to the plugin data dir OFF the trust-hashed profile surface; reads git metadata only (no repo code, no network); off the hook hot path (mined at bootstrap, read at Stop). Advisory only, never a block; fails open. The consumer re-checks each partner still exists before nudging. `=0` skips the mine and the advisory. See `plugin/mcp/chameleon_mcp/cochange_history.py`.
- `CHAMELEON_CANONICAL_GIT_RECENCY=0` ŌĆö kill switch for git-commit-time canonical recency (default ON). At bootstrap/refresh, canonical-witness selection ranks each cluster's files by a recency weight that decays off the file's LAST GIT COMMIT TIME (a single bounded `git log` walk per pass builds a `{path: commit_epoch}` map; smooth exponential decay, half-life `CHAMELEON_CANONICAL_RECENCY_HALF_LIFE_DAYS` default 45, walk timeout `CHAMELEON_CANONICAL_GIT_LOG_TIMEOUT_SECONDS` default 20). Commit time survives a fresh clone's uniform mtimes, so a recently-committed minority idiom outranks the legacy majority (the mid-migration case) instead of tying and falling to typicality. `=0` skips the git walk and uses the legacy mtime step (2.0 within 90 days, else 1.0). Off the hook hot path (bootstrap/refresh only); reads git metadata, never repo code or the network; fails open to the mtime step when git is unavailable, the tree is bare, a file is untracked, or the walk times out. See `_build_commit_time_map` / `_file_recency_weight` in `plugin/mcp/chameleon_mcp/bootstrap/canonical.py`.
- `CHAMELEON_INTERP_CACHE=0` ŌĆö kill switch for the interpreter-resolution cache (default ON). `plugin/hooks/_resolve-python.sh` persists the ladder's winning argv to `${CHAMELEON_PLUGIN_DATA:-$HOME/.local/share/chameleon}/interp.cache` (line 1 = the mcp_dir it was resolved for ŌĆö the installed plugin path embeds the version, so the cache is version-keyed for free; following lines = argv tokens) and serves later resolves from it, so every hook skips the ladder once SessionStart has warmed it. A hit requires the recorded mcp_dir to match AND the leading argv binary to still resolve (`[ -x ]`, or `command -v` for a bare name); a deleted venv or uninstalled uv is a miss that re-runs the ladder and atomically rewrites the entry (write tmp + `mv`). Corrupt/unreadable/mismatched cache = miss, never an error, and the hit path is pure shell builtins (no fork), so a warm per-edit resolve survives even a hostile PATH. With neither `CHAMELEON_PLUGIN_DATA` nor `HOME` set, caching is disabled entirely rather than falling back to a world-writable temp dir (a planted cache there could hand every hook an attacker-controlled interpreter argv). Local-only file I/O, no repo-code execution and no network, so it follows the default-ON-with-kill-switch principle; `=0` bypasses read AND write.
